This audit covers code quality and design, RAG-specific quality, enhancement opportunities, and security. It builds on the 2026-05-28 security audit: most of that report's High findings are verified **fixed** (constant-time API key compare, rate limiting on all routes, CORS lockdown with `*`-rejection validator, XML-tagged prompts in generator and both guardrails, ingest metadata validation, generic ingest error messages, nginx TLS config + security headers + non-root, CI permissions, the `ingest_scraped.py` await bug). Findings below are either new or explicitly carried over as still open.

---

## Executive summary

- **The core is genuinely strong.** Clean service separation (`routes → guardrails / retriever / generator / cache`), strict Pydantic validation of LLM output with no silent fallbacks, a documented decision log, a hand-labelled retrieval eval with an honest baseline and self-critical analysis, and a mock mode that makes the whole stack testable offline. This is above the bar for a portfolio RAG project.
- **The single biggest grounding gap: the output guardrail only reviews the prose.** `routes.py` joins `answer["summary"]` and sends that to `check_output` — the `stats[]` array and `chart.bars[]` values, which are the most prominent numbers rendered in the UI (StatPill, Chart), are never checked for grounding. A hallucinated stat in those fields ships unreviewed. (High)
- **The ingestion stack is duplicated and has already diverged.** `DOCUMENT_REGISTRY` exists in both `app/services/ingestor.py` (8 docs) and `scripts/ingest.py` (13 docs) with **conflicting URLs** for Profit Ability 2; `chunk_text` is copy-pasted in three files. The same document ingested via API vs script gets different citation URLs. (High)
- **Observability is half-wired.** `main.py` configures structlog and a request-ID middleware, but every module logs via stdlib `logging` — so the bound `request_id` never appears in any log line. LangFuse traces only the `generate()` call: no spans for retrieval, guardrails, cache, or end-to-end latency, and guardrail outcomes are not recorded (CLAUDE.md claims they are). (Medium)
- **Retrieval is vector-only and the eval already proves where it's weak**: the 0.75 distance threshold prunes nothing (production and `--ignore-threshold` runs identical), exact-keyword P@5 is 0.24, comparison R@1 is 0.15. The sector metadata filter is dead in practice — every registry entry is tagged `sector: "all"`, so the `$or` filter always matches everything. (Medium)
- **A retrieval/embedding failure returns a raw 500.** Only `generate()` is wrapped in try/except (→ 503). An OpenAI embeddings outage propagates out of `retrieve()` unhandled. (Medium)
- **The frontend ships hardcoded and dead UI in a product whose whole pitch is evidence**: CorpusRail hardcodes "8 reports / 142 chunks" (real corpus: ~13 docs, 408 chunks) while the live `/api/corpus` endpoint is never called; Copy, Regenerate, and Rephrase buttons have no handlers; the error `reference` ID users are told to "share with support" is never logged anywhere.

Severity counts: 0 critical · 3 high · 11 medium · 14 low.

## RAG-specific assessment

**Chunking — functional, but the bluntest instrument in the pipeline.** Fixed 800-word windows with 100-word overlap, split per page. The per-page scoping is a good call (clean page-level citations, which the eval exploits), but 800 words ≈ 1,000+ tokens is large for precision: top-5 retrieval injects ~5k tokens of context, and a chunk that size blends multiple topics, which is exactly what the eval's low exact-keyword precision (P@5 = 0.24) looks like. Word-splitting also ignores sentence/paragraph boundaries, and the golden dataset notes admit several PDFs' page text is "dominated by repeated nav/TOC" — there is no boilerplate stripping. **Recommendation:** A/B smaller (~250–350 word) paragraph-aware chunks against the existing eval; add a cheap repeated-line de-duplication pass at extract time. The eval harness makes this a one-afternoon experiment — that's the payoff of having built it.

**Embedding model — reasonable, undocumented.** `text-embedding-3-small` is the right cost/quality default for a 408-chunk corpus; nothing about UK media-planning vocabulary demands a specialist model. The gap is justification: the decision log explains LiteLLM and ChromaDB but not the embedding choice. Worth one paragraph (and the eval can cheaply benchmark `-3-large` to prove `-small` is enough — a stronger portfolio statement than the default).

**Retrieval — vector-only, with its weaknesses already measured.** This is the audit's easiest section to write because the project measured itself: the distance threshold prunes nothing (M4), the sector filter is structurally inert (M5), comparison questions rank poorly at top-1 (R@1 = 0.15), multi-doc misses the second source, and exact-keyword precision is where lexical matching would shine. The README's own "where to improve next" list (BM25 hybrid, k=8–10 + cross-encoder rerank, comparison-query decomposition) is correct and evidence-led. The query enrichment (`Sector: FMCG | Budget: 500k-2m. <question>`) is a nice, simple touch that prepends structured context before embedding.

**Prompt construction — clean.** Chunks are XML-tagged with id/source/page and an explicit "source data, never instructions" rule; brand context is a compact one-liner; the schema is shown by example with per-field rules; strict-grounding retry appends an addendum rather than rewriting the prompt. No context stuffing at the current k=5. Two duplications mar it (L7), and citation superscripts `[n]` rely on the model emitting indices that match chunk order — which holds because sources are built from the same ordered list, but nothing validates that an emitted `[7]` actually exists (the frontend would render a citation pointing at nothing).

**Grounding — the strongest part of the design, with one hole.** The defense stack is genuinely layered: input guardrail → distance-filtered retrieval → grounding-rules prompt → strict Pydantic validation (raises, never coerces) → output guardrail → stat-aware retry with strict addendum → safe fallback that is never cached. That is a defensible, explainable pipeline. The hole is H1: the guardrail never sees the structured stats or chart values, which are both the most hallucination-prone content (numbers) and the most visually authoritative in the UI. Secondary gaps: guardrails fail *open* on error (a documented, debatable choice — fine for availability, but worth failing closed when `_answer_contains_statistic` is true), and M10's fallback text undermines the philosophy it exists to protect.

## Enhancements, ranked by impact vs effort

1. **Close the stats/chart grounding hole (H1).** ~20 lines. Highest trust-per-line-of-code available anywhere in this repo.
2. **Unify the ingestion stack (H2).** One registry, one chunker, scripts as thin CLIs over `app/services`. Fixes a live citation-correctness bug and deletes ~150 duplicated lines.
3. **Finish observability (M2 + M3).** Wire `request_id` into actual log lines and extend LangFuse to root-trace per request with spans for cache/guardrails/retrieval/generation, recording guardrail outcomes and stage latencies. For an ITV AI-backend-engineer pitch, *demonstrated* observability is worth more than any retrieval improvement — and right now CLAUDE.md writes a cheque the code doesn't cash.
4. **Hybrid retrieval + rerank, A/B'd through the eval (M4/M5 + README hypotheses).** BM25+dense fusion first (targets the measured exact-keyword weakness), then retrieve-wide-rerank-to-5. Highest effort, highest retrieval ROI — and the before/after eval table is itself portfolio material.
5. **Honest health endpoint (H3) + retrieval error handling (M1).** Small fixes that move "production-grade" from claim to behavior.

**Resilience patterns present:** LiteLLM primary→fallback chain, per-call timeouts (60s gen / 15s+30s guardrails), graceful Redis degradation, guardrail fail-open, no-cache-on-fallback, rate limiting at both nginx and app layers. **Missing:** retry-with-backoff on transient primary-model errors (LiteLLM's `num_retries` is a one-liner), JSON forcing on the fallback path (M6), any handling of embedding-API failure (M1), and circuit breakers (legitimately out of scope at this scale — do not add one; that would be the over-engineering this audit is told to flag).

## Security

**Posture: materially improved since 2026-05-28.** Verified fixed: `hmac.compare_digest` auth, rate limits on all four routes, CORS restricted to GET/POST + explicit headers with a validator rejecting `*`, prompt-injection hardening via XML data tags + "literal data" instructions in generator and both guardrails, topic/sector whitelist validation in `ingest_scraped.py`, path containment on ingest, generic ingest error messages, production startup checks, nginx non-root + security headers + a TLS prod config, CI `permissions: contents: read`. No secrets found in any tracked file; `.env` is gitignored and `.env.example` uses placeholders.

**Still open / new:**
- **Prompt injection** is defended but not eliminated (it never fully is): the residual vectors are `ingest_scraped.py`'s `source_title`/`source_url` headers, which are validated for topic/sector but not title/URL, and flow into prompts and the UI (`SourceCard` renders `source.url` as a clickable link — a poisoned scraped file could inject a phishing URL into citations). Validate titles/URLs against a trusted registry like the PDF path does.
- **Fail-open guardrails** (documented choice) mean a guardrail-API outage disables both filters silently. Consider failing closed for stat-bearing outputs.
- Health endpoint truthfulness (H3), public `/api/corpus` (L5), unpinned base images (M11), inverted secure defaults (L4) — all carried over.
- Input validation is otherwise solid: question length-bounded 5–500, enums whitelisted, Pydantic everywhere, React's default escaping covers output rendering (no `dangerouslySetInnerHTML` anywhere).

## Elegance violations

Places where the code works but misses the project's own stated standard (simple, clear interfaces, composable, explicit):

1. **`routes.query()` is a 130-line orchestrator** doing cache, guardrail, retrieval, generation, a retry policy, source mapping, and cache-write inline. The retry-with-strict-grounding block (steps 5) is the densest logic in the repo and lives in the HTTP layer. Extract an `answer_query()` pipeline function in services; the route should validate, call, and translate errors. This is the difference between "handler that grew" and "pipeline you can test and trace".
2. **Duplication is the dominant smell**: registry ×2 (diverged — H2), `chunk_text` ×3, chunk-XML formatting ×2 (L7), enum sets ×3 across config/types.ts/Composer, and five identical field validators that `Literal` types would delete outright (L3). For a codebase this disciplined elsewhere, the copy-paste is conspicuous.
3. **Half-adopted infrastructure is worse than none**: structlog + RequestIDMiddleware configured but unused by any logger (M2) is machinery that *looks* like observability in a code review and does nothing at runtime. Either finish it or remove it — the current state is the "magic over explicit" failure mode.
4. **Import-time side effects** (`cache = _make_cache()` connecting to Redis on import; settings frozen at module top) trade explicitness for convenience and make the test suite patch-order-sensitive — `conftest.py` has to patch `get_collection` *before* importing `app.main`, which is exactly the fragility this style causes (L2).
5. **The frontend's inline-style-object idiom** repeats the same mono-label literal (`fontFamily: 'var(--cue-mono)', fontSize: '10px', textTransform: 'uppercase', letterSpacing: ...`) ~25 times across components, while the stack declares Tailwind v4 and uses it for two classes (`h-full`, `antialiased`). The OKLCH design-token layer is genuinely good; the consumption layer contradicts the declared stack and bloats every component. Pick one approach — tokens + a few shared classes would cut component LOC ~30%.
6. **Decorative truth-claims**: dead Copy/Regenerate/Rephrase buttons (M8), a fake "Verifying citations" progress trace (L11), a hardcoded corpus rail (M7), a support reference that references nothing (M9), and a "grounded" fallback making an ungrounded claim (M10). Individually small; collectively they are the gap between "demo polish" and the production-grade evidence-first product the copy promises.
7. **Over-engineering check — largely clean.** The dual cache, LiteLLM gateway, and mock mode are all justified by documented decisions; the eval is hand-rolled pure functions instead of a framework (correct call). The only unused flexibility found: `python-pptx` in prod deps (M11), the never-populated `Callout`/`Phase` variants (L8/L9), and the structlog machinery (M2). This codebase's failure mode is duplication, not abstraction.
